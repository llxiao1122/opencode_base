# Cipher 全架构审计报告

> 生成日期：2026-07-21 | 基于 v2.5 | 基于全源文件静态分析

---

## 首要发现

本审计基于 **110 个 `.py` 源文件（~25000 行）** 的静态分析。系统已从最初的"规则驱动办公自动化"演进至今，核心数据流清晰，但存在架构漂移、SSOT 违规、死代码和模糊的 LLM 边界。

## 3. LLM 调用全景（共 12 个调用点）

| 序号 | 调用位置 | 触发条件 | 输入 | 用途 | 风险等级 |
|------|----------|----------|------|------|----------|
| 1 | `entry.py:444` _handle_event | event 路径（未匹配 task/knowledge/profile） | event + context + 检索数据 | **回复合成** | 低 — 仅表达 |
| 2 | `entry.py:103` _cached_llm | knowledge_handler/profile_handler 调用 | 知识片段 + 用户查询 | **回答合成** | 低 — 仅表达 |

### LLM 调用路径图

```
                reasoning/llm_client.call()  ← 唯一 LLM API 封装
                         │
        ┌────────────────┼────────────────────┬──────────────┐
        │                │                    │              │
   entry._cached_llm  composer._llm       llm_tags      memory_core
        │                │                    │              │
   ┌────┼────┐          │                    │          nightly.py
   │    │    │     execute_plan()       extract_tags   reflect()
   │    │    │          │                    │
knowledge profile _handle_event         route_request
handler handler     │                        (废弃)
                    │
              _llm() direct call
                 → 回复合成
```

### 结论：当前 12 个 LLM 调用点中

- **7 个是"表达合成"**（1-6）—— 低风险，按设计
- **1 个是"语义标签提取"**（7）—— 中风险，受限于固定标签集
- **1 个是"经验提炼"**（8）—— **高风险**，LLM 输出直接写入长期记忆
- **2 个是"因果推演"**（9-10）—— 仅探索，不影响生产
- **1 个是"导入补充"**（11）—— 仅批量导入
- **1 个是"偏好提取"**（12）—— 仅更新偏好文件

### 4.1 SSOT 违规

| **组织数据** | `state/org.md` 和 `organization/model.py` | org.md 仅用于 `hierarchy_resolver`，`OrganizationModel` 另有独立硬编码字典 |

### 4.2 AGENTS.md 约束被违反

**"不让 composer 决定任务"** — `composer.py:95` 中 `execute_plan` 确实不直接调用 `task_handler`（当前架构中 `composer` 未被 `entry.py` 使用），但 `composer` 保留了对 task 路径的编排能力。

**"不让 prompt 承担架构逻辑"** — 多处 prompt 包含"不替用户决策"、"基于数据分析"等架构级约束（`entry.py:420-423`, `composer.py:115-120`），说明架构决策已泄漏到 prompt 中。

**"不让 LLM 判断责任归属"** — 当前代码中 `core/context.resolve()` 用纯规则判断责任，**本次审计未发现违反**。但 `_handle_event` 的 sys_prompt 包含"不替用户决策"——说明设计者担心 LLM 越界。

### 关键发现：循环依赖

```
entry.py → profile_handler.py → _cached_llm → entry.py
entry.py → knowledge_handler.py → _cached_llm → entry.py
```

## 7. 关键建议总结

### 必须改（优先级 P0）

1. **移除 `_cached_llm` 的循环依赖**：将缓存逻辑移到 `reasoning/llm_client.py` 或 `tools/shared/` 中
2. **删除或标记废弃模块**：`slow_think.py`, `simulator.py`, `value_arbiter.py`, `curiosity_engine.py`, `route_request.py`, `llm_tags.py`, `composer.py`, `context_builder.py`, `plugins/*` — 这些模块共计约 1250 行代码，无人调用
3. **修复 SSOT**：`TEAM_MAP` 和 `OrganizationModel._teams` 冗余。统一从 `state/` 目录加载

### 建议改（优先级 P1）

4. **消除 Memory 层的 LLM 风险**：`memory_core.reflect()` 的 LLM 输出直接写入 `memory/observations/`，应添加事实/模式/结论三层分开存储
